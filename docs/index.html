Anatomy of a
framing codec

TCP hands you a formless stream of bytes. To get discrete messages back out, you frame them: a length, then a payload. This page lets you drive the bytes yourself and watch the decoder make every decision — the two hard parts, partial reads and zero-copy, made visible.

length : u32 4 bytes · big-endian
payload exactly length bytes

Read four bytes to learn the size. Read that many more to get the message. That single idea — repeated correctly under messy network conditions — is the whole protocol.

01

TCP has no message boundaries

why framing has to exist at all

TCP guarantees your bytes arrive in order and intact. It guarantees nothing about grouping. Three write() calls can land in the reader as one big chunk, or be sliced at arbitrary points. The boundaries you sent with are gone the moment the bytes hit the wire.

The dashed red edges are where a single read() happened to cut. Message B got split across two reads; C got glued onto the tail of B's. Order preserved, boundaries lost. A length prefix is how you put the boundaries back.

Notice the pattern: the decoder never destroys buffered bytes it can't yet use. A half-arrived frame just sits in the buffer, and the reserve hint tells the transport roughly how many more bytes to go fetch. That's the entire trick to surviving arbitrary network fragmentation.

03

Hard part one — partial frames

correctness under fragmentation

The naive decoder assumes one read equals one message. It passes every test on localhost, where messages usually do arrive whole — then quietly corrupts data in production the first time the network splits a frame. The fix is a decoder that treats "not enough bytes yet" as a normal, expected state, not an error.

  • 1.Prefix not complete? If fewer than 4 bytes are buffered, you can't even know the frame size. Return Ok(None), keep the bytes, reserve room for the rest of the prefix. (Try the Drip 1 byte button above — it stays Ok(None) for the first three bytes.)
  • 2.Prefix complete, payload not? Now you know the length, but the body is still arriving. Return Ok(None) again and reserve exactly the number of missing bytes — so the next read can land in one allocation instead of growing the buffer repeatedly.
  • 3.Whole frame present? Only now do you consume bytes and hand out a frame — then immediately loop and try to decode the next one, because a single read may hold several (the "two frames glued" scenario).

The load-bearing detail is what doesn't happen: no partial data is ever dropped or misread. The buffer is the decoder's memory between calls.

04

Hard part two — zero-copy payloads

performance on the hot path

Once a whole frame is buffered, how do you hand it to the application? The obvious way allocates a fresh Vec<u8> and copies the payload into it. Fine once; at hundreds of thousands of messages per second it becomes allocator pressure and cache traffic — pure overhead on the busiest path in the program.

05

The decode() function, annotated

every branch you just watched, in ~20 lines

Here is the real thing from src/codec.rs. The colored highlights map to the simulator: prefix checks, the oversized guard, the payload check, and the zero-copy handoff.

fn decode(&mut self, src: &mut BytesMut) -> Result<Option<Bytes>> {
    // (1) Not even the length prefix yet — wait for more.
    if src.len() < 4 {
        src.reserve(4 - src.len());   // hint: how much more we want
        return Ok(None);
    }

    // Peek the length without consuming — a partial payload must
    // leave the buffer untouched for the next call.
    let payload_len = u32::from_be_bytes(src[..4].try_into()?) as usize;

    // (2) Reject oversized frames BEFORE allocating anything.
    if payload_len > self.max_frame_len {
        return Err(FrameError::FrameTooLarge { .. });
    }

    // (3) Prefix is here, but the whole payload isn't yet.
    if src.len() < 4 + payload_len {
        src.reserve(4 + payload_len - src.len());
        return Ok(None);
    }

    src.advance(4);  // drop the prefix; we don't need it anymore

    // (4) Zero-copy handoff: split_to detaches the payload into a
    // handle sharing the SAME allocation; freeze makes it immutable.
    // No per-frame Vec, no memcpy — just a refcount bump.
    let payload = src.split_to(payload_len).freeze();
    Ok(Some(payload))
}

Four guards, in order: prefix → size limit → payload → emit. Each early return is either "wait for more" or "reject". Only the final two lines produce a frame, and they don't copy it.

06

The model to keep

what to remember when the code is gone

Where does state live between reads?
In the BytesMut buffer itself. The decoder is a pure function of "bytes currently buffered". Incomplete data simply stays there until the rest arrives — the buffer is the memory.
What does Ok(None) actually mean?
Not an error — a normal "insufficient bytes, try again later". It's the single mechanism that makes the decoder immune to how the network chose to chunk the stream.
Why is the payload a Bytes, not a Vec?
Bytes is a refcounted view into the buffer that already holds the data. Handing it out costs a refcount bump, not an allocation and a copy. The shared buffer frees itself when the last view drops.
Is this what real systems do?
Yes — Postgres, Redis, Kafka, gRPC and game/RTC servers all frame length-first over TCP. They add auth, versioning, multiplexing on top, but these two mechanics — framing under partial reads, zero-copy payloads — are the same spine.
